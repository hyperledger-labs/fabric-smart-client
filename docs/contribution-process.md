# Contribution Process

This document defines the expected workflow for contributing to this repository.

The goal is to keep contributions simple, well-scoped, and easy to review.

---

## Getting Started

* Browse open issues
* Comment on the issue to express interest
* Wait for maintainer confirmation before starting work

> For non-trivial changes, discussing the approach in the issue is recommended before implementation.

---

## Development

* Create a branch:

  ```
  feat/issue-<id>-short-description
  ```
* Keep changes focused on the issue
* Avoid unrelated refactoring or large unscoped changes

---

## Pull Requests

* Open a PR against `main`
* Include:

  * Issue reference (e.g., `Closes #123`)
  * Clear description of the change
* Prefer small, incremental PRs over large changes

---

## Review

* Maintainers will review the PR
* Address feedback promptly
* Keep discussions focused on the proposed change

---

## Merge

* PRs are merged after approval
* Referenced issues are closed automatically

---

## Guidelines

* Discuss larger design changes before implementation
* Keep PRs minimal and easy to review
* Align with existing code structure and style

---

## Summary

This process ensures contributions remain consistent, reviewable, and aligned with the project.
